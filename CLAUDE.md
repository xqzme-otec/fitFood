# Требования к функционалу (реализовать полностью)

## 1. Аутентификация и профиль
- Регистрация / логин (JWT или сессии).
- После первого входа — анкета:
  - пол, рост (см), вес (кг), возраст (лет), средняя дневная активность (минимальная/низкая/средняя/высокая/очень высокая).
  - цель: «набор массы», «поддержание», «похудение».
  - Если цель «набор» или «похудение» — дополнительные поля: целевой вес (кг) и количество дней для достижения цели.
- Пользователь может в любой момент **изменить текущий вес** (отдельный эндпоинт). Это пересчитывает его норму КБЖУ.
- Пользователь может **изменить дневной лимит калорий** вручную — суточные БЖУ должны автоматически пересчитаться пропорционально новому лимиту (с сохранением соотношения белков/жиров/углеводов, рассчитанного по его цели).

## 2. Расчет КБЖУ и режим питания
- На основе анкеты и цели рассчитать:
  - базовый обмен веществ (BMR) по формуле Миффлина-Сан Жеора.
  - суточную норму калорий с поправкой на активность и цель (дефицит/профицит/поддержание).
  - суточные нормы белка (г), жиров (г), углеводов (г), исходя из цели и предпочтений (например, для набора массы белок 1.8-2.2 г/кг веса, для похудения — 1.6-2.0 г/кг).
- Пользователь указывает **количество приёмов пищи в день** (например, 3: завтрак, обед, ужин).
- Для каждого приёма задаётся **лимит калорий** (в сумме равный суточной норме). Можно задать фиксированные проценты (30/40/30) или пользователь вводит вручную. Лимиты БЖУ на приём — пропорционально калориям приёма.
- За каждый приём пользователь добавляет съеденные продукты/блюда. Система суммирует КБЖУ за день и по приёмам, показывает остаток лимитов.

## 3. База продуктов и блюд
- Таблицы: `products` (id, название, белки, жиры, углеводы, калории на 100 г, единица измерения, категория), `dishes` (блюда, состав из продуктов с весом).
- Минимальная предустановленная база: борщ, кесадилья с курицей, сыр, помидоры, макароны, молоко и т.д. (не менее 20 продуктов и 5 блюд).
- Пользователь может искать продукт/блюдо через поиск, указывать съеденное количество (г или мл), и оно записывается в конкретный приём пищи (завтрак/обед/ужин). Дневное КБЖУ пересчитывается.

## 4. Умный холодильник
- Отдельный раздел «Мой холодильник». Продукты группируются по категориям: крупы, овощи, фрукты, мясо, рыба, молочка, соусы, прочее.
- Добавление продуктов:
  - **Вручную**: название, количество (г/мл/шт), срок годности (дата) — если не указана, LLM предлагает примерную дату.
  - **Сканированием чека**: пользователь загружает фото чека или вводит текст. Система через OCR + LLM выделяет **только продукты питания** (мыло, бытовая химия отбрасываются). Для каждого распознанного продукта берётся название, количество, цена (опционально), срок годности (если есть на чеке; если нет — LLM генерирует разумный срок хранения на основе типа продукта).
- Функции управления:
  - изменение количества продукта (например, макароны 500 г → 100 г после использования).
  - удаление продукта.
  - отображение срока годности с визуальным предупреждением (скоро истекает/просрочен).
- Все данные холодильника привязаны к пользователю.

## 5. Рекомендации рецептов (на основе содержимого холодильника и КБЖУ)
- Система анализирует:
  - оставшиеся лимиты КБЖУ на текущий приём пищи (например, ужин).
  - список реально доступных продуктов в холодильнике.
- Из базы рецептов выбираются блюда, которые:
  - можно приготовить из имеющихся продуктов (допускается отсутствие неключевых ингредиентов — но с пометкой).
  - по своим КБЖУ покрывают дефицит или соответствуют цели пользователя (например, не хватает белка → рекомендация блюда с высоким содержанием белка).
- Если ни одно блюдо не подходит, предлагаются простые комбинации продуктов (например, творог с ягодами).
- Пользователь может запросить рекомендацию для любого приёма пищи или «на сейчас».

## 6. Интеграция с LLM (для распознавания чеков и определения сроков годности)
- Использовать внешний API (OpenAI, локальная модель через Ollama или другой сервис).
- Для чека: передать текст OCR (или само изображение) с промптом: *«Извлеки из чека только пищевые продукты. Верни список: название, количество, срок годности (если указан). Непродукты (мыло, салфетки и т.д.) игнорируй. Для продуктов без срока годности предложи типичный срок хранения»*.
- Для ручного добавления: если пользователь не указал срок годности, вызвать LLM для оценки: *«Для продукта "X" укажи средний срок годности (в днях) при хранении в холодильнике/комнатной температуре. Ответ только числом»*.
- Уже есть xgb_model.pkl для классификации категории продуктов в холодильник всего классов(Готовая еда, Молочный прилавок, Хлеб и выпечка, Овощи и фрукты, Бакалея, Консервы, Мясо птица рыба, Рыба морепродукты, Замороженные продукты, Сладости, Снеки, Чай кофе какао, Вода и напитки)
- Так же при скане чека пользователю должен быть показан список для подтверждения того что он добавляет в холодильник, например он купил мыло брокколи и колбасу, после скана чека пользователю должно быть показано окно что мыло-отброшено, брокколи-Овощи фрукты, колбаса-мясо птица рыба он подтверждает и все добавляется в холодильник
